# ЛОГРУС MVP: верхнеуровневый план реализации

Дата фиксации: 2026-06-17  
Статус: high-level roadmap. Документ предназначен для последующей детализации в задачи, спринты и технические спецификации.

## 1. Цель MVP

Проверить, что ЛОГРУС может стабильно собирать открытые события по теме мобилизационных и военно-экономических рисков для бизнеса, связывать их между собой и выдавать проверяемую аналитическую сводку с основаниями и первоисточниками.

Ключевая проверяемая цепочка:

```text
источник -> документ/материал -> событие -> связь -> основание -> вывод -> пользовательская проверка
```

MVP не должен решать все будущие задачи платформы. Главная цель — доказать ценность модели "событие -> связь -> вывод" на узкой теме.

## 2. Принципы реализации

- Начинать с модульного монолита и async worker, без микросервисов.
- Не хардкодить мобилизационную тему в ядро: тема должна жить в конфигурации.
- Сначала подключить источники P0, затем расширять контур.
- Разделять факты, тематические связи и аналитические гипотезы.
- Всегда хранить ссылку на первоисточник и evidence.
- LLM использовать как помощника извлечения и объяснения, а не как источник истины.
- Для важных или низкоуверенных выводов предусмотреть ручную проверку.
- Делать продукт вокруг карточки события и оснований, а не вокруг сложного графа.

## 3. Фаза 0. Продуктовая рамка и критерии успеха

Цель: зафиксировать, что именно считаем успешным MVP.

Работы:

- выбрать 1 основную тему MVP: мобилизационные и военно-экономические риски для бизнеса;
- определить 2-3 подтемы для первой версии;
- выбрать 5-10 потенциальных пользователей для реалити-чека;
- определить формат первой демонстрации: daily brief, weekly brief или интерактивный dashboard;
- зафиксировать метрики MVP.

Результаты:

- список подтем;
- список первых пользователей/экспертов;
- критерии успешной демонстрации;
- decision log по ограничениям MVP.

Пример критериев успеха:

- система собирает события минимум из 4 источников P0;
- на историческом периоде собрано 50-100 событий;
- для ключевых событий есть первоисточник;
- минимум 10-20 связей имеют объяснение и evidence;
- пользователь понимает, почему событие важно, без устного объяснения автора продукта.

## 4. Фаза 1. Технический фундамент

Детальный документ фазы:

[logrus-phase-1-technical-foundation.md](phase-1/logrus-phase-1-technical-foundation.md)

Цель: поднять минимальный backend, storage и рабочий контур разработки.

Работы:

- выбрать стек MVP;
- создать backend/API;
- создать worker-процесс для фоновой обработки;
- поднять PostgreSQL + pgvector;
- создать raw store для исходных HTML/XML/PDF/text snapshots;
- настроить миграции;
- добавить базовое логирование;
- добавить конфигурацию тем и источников;
- завести базовые роли: пользователь и аналитик.

Результаты:

- запускаемый локально MVP skeleton;
- первичная схема БД;
- конфиг темы MVP;
- базовый API health/status;
- минимальный observability слой.

Decision gate:

- система локально запускается одной командой;
- можно создать тему, источник и пустое событие через API/скрипт.

## 5. Фаза 2. Сбор данных P0

Цель: подключить первые источники и начать регулярно получать сырые материалы.

Приоритет источников:

1. `publication.pravo.gov.ru`;
2. `regulation.gov.ru`;
3. `api.duma.gov.ru` / СОЗД;
4. `government.ru`;
5. `kremlin.ru`;
6. `mintrud.gov.ru`;
7. `trudvsem.ru`.

Работы:

- реализовать общий интерфейс source connector;
- подключить 2-3 самых надежных источника первыми;
- сохранять сырой снимок материала;
- сохранять метаданные: URL, дата публикации, источник, тип материала;
- добавить дедупликацию по URL/source_id/content hash;
- добавить расписание сбора;
- вести журнал ошибок по источникам.

Результаты:

- работающие коннекторы P0 первой волны;
- таблицы `sources`, `documents/materials`, `raw_snapshots`;
- регулярный сбор новых материалов;
- ручной backfill за выбранный исторический период.

Decision gate:

- за исторический период собрано достаточно материалов для 50-100 событий;
- повторный запуск не создает дубликаты;
- для каждого материала сохраняется первоисточник.

## 6. Фаза 3. Нормализация и извлечение событий

Цель: превратить сырые материалы в структурированные события.

Работы:

- реализовать parser/text extractor для HTML/XML/PDF/text;
- привести материалы к единому формату;
- определить минимальную схему события;
- сделать rule-based фильтр по теме;
- подключить LLM для извлечения событий из сложных текстов;
- извлекать акторов, ведомства, регионы, документы, цитаты и темы;
- сохранять уровень уверенности разметки;
- добавить review queue для спорных событий.

Результаты:

- события создаются из материалов;
- событие содержит источник, дату, тип, краткое описание, темы и evidence;
- спорные события отправляются на ручную проверку.

Decision gate:

- на тестовом наборе большинство релевантных материалов превращается в события;
- нерелевантный шум не проходит в основную сводку;
- аналитик может поправить событие вручную.

## 7. Фаза 4. Связи и гипотезы

Цель: реализовать минимальную версию главной магии продукта.

Работы:

- зафиксировать разделение прямых ground-связей и связей событие-событие:
  - `event_entities` хранит акторов, ведомства, регионы и другие сущности события;
  - `event_topics` хранит принадлежность события к теме;
  - `documents` и `raw_snapshots` хранят происхождение события;
  - `relations` хранит только связи событие-событие.
- реализовать фактические связи событие-событие:
  - общий документ;
  - ссылка на документ;
  - стадия законопроекта;
  - версия/дубликат материала.
- реализовать тематические связи:
  - общая тема;
  - близкие даты;
  - похожие формулировки;
  - общий регион;
  - институциональный контекст.
- использовать общий актор, ведомство и временную близость как сигналы скоринга, а не как самостоятельные причины создать связь;
- добавить `relation_topic_scores`;
- добавить evidence для каждой связи;
- добавить объяснение связи;
- реализовать первые аналитические гипотезы поверх цепочек событий;
- добавить ручную проверку для важных гипотез.

Результаты:

- таблицы/модели `relations`, `relation_topic_scores`, `evidence`, `analytical_hypotheses`;
- связи событие-событие разделены на фактические, тематические и гипотезы;
- карточка события показывает прямые факты события отдельно от связанных событий и оснований связи.

Decision gate:

- пользователь видит разницу между фактической связью, тематической связью и гипотезой;
- каждая важная связь раскрывается до evidence;
- `PRECEDES` не используется как самостоятельный тип связи;
- гипотеза не выглядит как установленный факт.

## 8. Фаза 5. Аналитическая сводка

Цель: сформировать первый полезный выход продукта.

Работы:

- выбрать формат: daily brief или weekly brief;
- отбирать важные события за период;
- группировать события по подтемам;
- показывать связи и основания;
- формировать краткие выводы через LLM;
- добавлять контраргументы и уровень уверенности;
- сохранять сгенерированные сводки;
- дать аналитику возможность править сводку перед отправкой.

Результаты:

- первая регулярная сводка;
- карточки событий внутри сводки;
- ссылки на первоисточники;
- ручная правка перед публикацией.

Decision gate:

- сводку можно показать внешнему эксперту;
- эксперт понимает ценность без доступа к внутренним данным;
- в сводке нет выводов без первоисточников.

## 9. Фаза 6. Минимальный продуктовый интерфейс

Цель: дать пользователю рабочий экран MVP.

Работы:

- реализовать экран сводки;
- реализовать список главных событий;
- реализовать карточку события;
- показать основания и первоисточники;
- показать связанные события;
- показать прямые факты события отдельно от связей с другими событиями;
- показать тип связи событие-событие: фактическая, тематическая, гипотеза;
- добавить фильтр по подтемам;
- добавить базовый поиск;
- добавить минимальные роли пользователя и аналитика.

Результаты:

- web UI по мотивам `logrus-mvp-home.svg`;
- пользователь может открыть сводку, событие, связь и первоисточник;
- аналитик может проверить или поправить событие/связь.

Decision gate:

- пользователь проходит основной сценарий без помощи разработчика;
- интерфейс не перегружен графами и лишними KPI;
- главная ценность видна через карточку события.

## 10. Фаза 7. Качество, eval и безопасность

Цель: снизить риск ложных выводов и модельных ошибок.

Работы:

- собрать eval-набор из реальных материалов;
- вручную разметить события и связи;
- измерять качество извлечения событий;
- измерять качество тематической фильтрации;
- проверять галлюцинации LLM;
- логировать стоимость LLM;
- настроить лимиты и кэширование;
- проверить отсутствие персональных данных;
- сформулировать дисклеймеры и границы продукта.

Результаты:

- первый eval-набор;
- метрики качества;
- список известных ограничений;
- правила безопасной генерации аналитических выводов.

Decision gate:

- ошибки понятны и отслеживаются;
- важные выводы проходят human review;
- продукт не позиционируется как инструмент обхода законодательства или инвестиционные рекомендации.

## 11. Фаза 8. Закрытый Preview

Цель: показать MVP ограниченному кругу пользователей и проверить реальную ценность.

Работы:

- выбрать 3-5 дизайн-партнеров или экспертов;
- настроить для них 1-2 подтемы;
- дать доступ к сводкам и карточкам событий;
- собрать обратную связь;
- измерить, какие события они открывают и сохраняют;
- провести интервью после использования;
- выделить повторяющиеся запросы и непонятные места.

Результаты:

- feedback report;
- список продуктовых правок;
- список недостающих источников;
- уточненный ICP;
- решение: продолжать, сузить тему или сменить первую вертикаль.

Decision gate:

- хотя бы часть пользователей говорит, что сервис экономит время или показывает связи, которые они бы искали вручную;
- понятно, за какой результат они потенциально готовы платить.

## 12. Что не входит в первый MVP

- полноценная graph database;
- enterprise SSO;
- API для клиентов;
- сложная BI-аналитика;
- автоматические инвестиционные рекомендации;
- массовый B2C;
- обработка закрытых или персональных данных;
- полный контур госзакупок без отдельного spike;
- private/on-prem deployment;
- большой набор тем за пределами первого фокуса.

## 13. Основные зависимости и неизвестные

- качество и стабильность API/RSS/HTML источников;
- доступ к API Госдумы и его актуальная работоспособность;
- сложность парсинга официальных документов;
- качество LLM на русском бюрократическом и правовом тексте;
- достаточность событий в выбранной теме для убедительного MVP;
- готовность пользователей доверять выводам при наличии evidence;
- юридическая безопасность позиционирования и бренда.
